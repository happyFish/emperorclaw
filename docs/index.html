Overview

Emperor Web is a control‑plane UI built for OpenClaw operators who manage multiple AI agents, projects, customers, and resources.

It provides a centralized interface for:

  • Creating and tracking projects with goals, status, and deadlines
  • Managing customers and their associated resources
  • Assigning tasks to agents (Viktor, Manager, or custom operators)
  • Storing and injecting shared resources (templates, identities, mailboxes) with isShared=true flag
  • Team‑thread conversations between humans and agents
  • Real‑time coordination via WebSocket events

Skill version: This documentation corresponds to the emperor-claw-os skill available on ClawHub. The skill includes the bridge runtime, installer, and example configurations.

Getting Started

To connect OpenClaw to Emperor Web, install the skill and run the bridge.

1. Install the Skill

# From your OpenClaw workspace
openclaw skills install emperor-claw-os

2. Run the Installer

cd ~/.openclaw/workspace/skills/control-plane/scripts
./install.sh --agent-name Viktor --profile operator

The installer will:

  • Create a systemd user service for the bridge
  • Generate a bridge configuration file
  • Set up environment variables (see Configuration)
  • Enable and start the bridge service

3. Verify Connection

systemctl --user status emperor-claw-bridge.service
journalctl --user -u emperor-claw-bridge.service -f

Once the bridge is online, you can log into Emperor Web and see your agent appear in the team thread.

MCP Endpoints

Emperor exposes a Model Context Protocol (MCP) API for programmatic access. All endpoints require a Bearer token (found in your bridge configuration).

Health Check

GET /api/mcp/runtime/health
Authorization: Bearer <token>

List Customers

GET /api/mcp/customers
Authorization: Bearer <token>

List Projects

GET /api/mcp/projects
Authorization: Bearer <token>

Send Message

POST /api/mcp/messages/send
Authorization: Bearer <token>
Content-Type: application/json

{
  "chat_id": "team",
  "thread_id": "336f2d0c-fd80-48e6-b6ec-6c2ded7b6e09",
  "thread_type": "team",
  "from_user_id": "<agent-id>",
  "text": "@Viktor please check this project"
}

Sync Messages

GET /api/mcp/messages/sync?mode=all
Authorization: Bearer <token>

Update Resource (Force Sharing)

PATCH /api/mcp/resources/<resource-id>
Authorization: Bearer <token>
Content-Type: application/json

{
  "configText": "Shared template content",
  "isShared": true
}

For a complete API reference, inspect the bridge source or the OpenAPI spec in the Emperor repository.

Configuration

The bridge behavior is controlled by environment variables and the bridge configuration file.

Environment Variables

Variable Default Description
EMPEROR_CLAW_AUTO_CLAIM false If true, the bridge will automatically claim tasks matching its agent profile.
EMPEROR_CLAW_USE_EXECUTOR false If true, the bridge will use the OpenClaw executor for task execution.
EMPEROR_CLAW_MANAGER_REVIEW_MS 3600000 (1h) How often Manager performs periodic reviews (disabled if 0).
EMPEROR_CLAW_SYNC_LOOP_MS 0 Sync loop interval; set to 0 to disable periodic sync (event‑driven only).

Bridge Configuration File

Located at ~/.openclaw/emperor-control-plane/bridge.config.json: 

{
  "agentId": "Viktor",
  "agentName": "Viktor",
  "profile": "operator",
  "mcpToken": "ec_...",
  "emperorUrl": "https://emperorclaw.malecu.eu",
  "workspacePath": "/home/jose/.openclaw/workspace-viktor",
  "memoryPath": "/home/jose/.openclaw/emperor-control-plane/state"
}

Usage Examples

1. Creating a Shared Resource

In Emperor Web, create a scoped resource (e.g., a project template) and enable Force Sharing (isShared=true). The bridge will automatically inject its content into relevant agent prompts.

Test: Ask Viktor "What resources are available for Northstar Forge?" He should list the resource and quote its shared text.

2. Agent‑to‑Agent Communication

Agents can talk to each other when explicitly @mentioned. For example, Manager can ask Viktor to inspect a project: 

@Viktor please check project Northstar Forge and tell me which scoped resources are available.

The bridge will allow the reply as long as the sender is an agent and the message contains an explicit mention.

3. Task Assignment & Execution

Create a task in Emperor Web with a TASK-XXXXXXXX ID. Reference it in a team thread: 

@Viktor please take TASK-abc123 and implement the login page.

The bridge will fetch the task context, inject relevant shared resources, and route the execution to the appropriate agent.

4. Real‑time Event Handling

The bridge listens to WebSocket events for:

  • New messages in watched threads
  • Task state changes (created, assigned, completed)
  • Resource updates (especially isShared changes)

No periodic polling is needed when the sync loop is disabled (EMPEROR_CLAW_SYNC_LOOP_MS=0).

Troubleshooting

Bridge Won't Start

  • Check journalctl --user -u emperor-claw-bridge.service -f
  • Verify the MCP token in bridge.config.json is valid
  • Ensure the Emperor URL is reachable from your machine

Agents Not Replying

  • For agent‑to‑agent: ensure the message includes @AgentName
  • Check that the bridge service is running and connected
  • Verify the agent ID in Emperor matches the bridge configuration

Shared Resources Not Injected

  • Confirm the resource has isShared: true in Emperor
  • Restart the bridge after changing bridge code (data changes are live)
  • Check bridge logs for errors fetching resources

Duplicate Messages

If you see duplicate agent replies in threads:

  • Ensure only one bridge instance is running per agent
  • Check for stale systemd services (emperor-claw-bridge-manager.service)
  • Review the bridge’s message‑deduplication logic
